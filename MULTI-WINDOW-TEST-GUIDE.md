# 多窗口屏幕共享测试指南

## 测试前准备

### 1. 启动服务

确保以下服务正在运行：

```bash
# 后端服务（端口 6099）
cd backend
mvn spring-boot:run

# WebSocket 服务（端口 6098）
# 应该已经包含在后端服务中

# 前端开发服务器（端口 3000）
cd frontend
npm run dev

# Electron 应用
cd frontend
npm run electron:dev
```

### 2. 准备测试环境

- 至少需要 2 个用户账号进行测试
- 建议使用两台电脑或两个浏览器窗口
- 确保摄像头和麦克风权限已授予

## 测试流程

### Phase 1: 基础窗口创建测试

#### 测试步骤：

1. **登录并创建会议**
   - 用户 A 登录系统
   - 创建一个新会议或快速会议
   - 进入会议页面

2. **开始屏幕共享**
   - 点击底部控制栏的"共享屏幕"按钮
   - 应该弹出屏幕共享选项对话框
   - 选择选项（可选：同时共享电脑声音、人像画中画）
   - 点击"开始共享"

3. **选择共享源**
   - 应该弹出 Electron 的屏幕源选择对话框
   - 可以看到：
     - "整个屏幕"标签页（显示所有屏幕）
     - "窗口"标签页（显示所有窗口）
   - 选择一个屏幕或窗口
   - 点击确认

4. **验证窗口创建**
   
   应该看到以下变化：
   
   ✅ **主窗口隐藏**
   - Electron 主窗口应该消失或最小化
   
   ✅ **顶部指示条出现**
   - 位置：屏幕最顶部
   - 内容：
     - 红点动画 + "您正在共享屏幕"
     - 会议时长（格式：MM:SS 或 HH:MM:SS）
   - 高度：40px（收起状态）
   
   ✅ **视频窗口出现**
   - 位置：右上角（默认）
   - 尺寸：300x400px
   - 内容：
     - 标题："会议成员 (1)" 或更多
     - 自己的视频/头像（绿色边框）
     - 音频控制按钮
   
   ✅ **聊天窗口出现**
   - 位置：左下角（默认）
   - 尺寸：350x500px
   - 内容：
     - 标题："聊天 (0)" 或消息数
     - 消息列表（可能为空）
     - 输入框和发送按钮
   
   ✅ **边框窗口出现（4个）**
   - 位置：共享区域的四个角
   - 尺寸：每个 60x60px
   - 外观：绿色 L 形边框，带发光效果

#### 预期结果：

- ✅ 所有 5 个窗口（1个主窗口隐藏 + 4个悬浮窗口）正确创建
- ✅ 窗口位置正确
- ✅ 窗口始终在最上层
- ✅ 边框不响应鼠标事件

#### 常见问题：

❌ **窗口没有出现**
- 检查控制台是否有错误
- 检查 Electron 主进程日志
- 确认 `window.electron.startScreenShareOverlay` 方法存在

❌ **窗口位置不正确**
- 检查 `main.js` 中的窗口创建位置计算
- 检查屏幕分辨率和工作区域

❌ **边框不可见**
- 检查 SVG 绘制逻辑
- 检查 corner 参数是否正确传递

### Phase 2: 交互功能测试

#### 测试 2.1: 顶部指示条交互

1. **鼠标悬停展开**
   - 将鼠标移到屏幕顶部
   - 指示条应该从 40px 展开到 80px
   - 应该显示控制按钮：
     - 静音/解除静音
     - 开启/关闭视频
     - 成员
     - 聊天
     - 暂停/恢复共享
     - 结束共享（红色）

2. **鼠标移开收起**
   - 将鼠标移开顶部
   - 指示条应该收起到 40px
   - 控制按钮应该隐藏

3. **点击控制按钮**
   - 点击"静音"按钮
     - 按钮应该变为激活状态（红色）
     - 图标应该变为 🔇
   - 点击"关闭视频"按钮
     - 按钮应该变为激活状态
     - 图标应该变为 🚫
   - 点击其他按钮，验证功能

#### 预期结果：

- ✅ 悬停展开/收起动画流畅
- ✅ 按钮点击有响应
- ✅ 按钮状态正确更新
- ✅ 操作发送到主窗口

#### 测试 2.2: 视频窗口交互

1. **拖动窗口**
   - 在窗口标题栏按住鼠标左键
   - 拖动窗口到不同位置
   - 释放鼠标

2. **折叠/展开窗口**
   - 点击标题栏的 "−" 按钮
   - 窗口应该折叠，只显示标题栏
   - 再次点击 "+" 按钮
   - 窗口应该展开，显示内容

3. **音频控制**
   - 点击自己视频上的音频按钮
   - 应该切换静音状态

#### 预期结果：

- ✅ 窗口可以拖动到任意位置
- ✅ 拖动过程流畅，无卡顿
- ✅ 折叠/展开动画正常
- ✅ 音频控制功能正常

#### 测试 2.3: 聊天窗口交互

1. **拖动窗口**
   - 在窗口标题栏按住鼠标左键
   - 拖动窗口到不同位置
   - 释放鼠标

2. **发送消息**
   - 在输入框输入消息
   - 点击"发送"按钮或按 Enter
   - 消息应该出现在消息列表中

3. **折叠/展开窗口**
   - 点击标题栏的 "−" 按钮
   - 窗口应该折叠
   - 再次点击 "+" 按钮
   - 窗口应该展开

#### 预期结果：

- ✅ 窗口可以拖动
- ✅ 消息可以发送
- ✅ 消息显示在列表中
- ✅ 折叠/展开正常

### Phase 3: 数据同步测试

#### 测试 3.1: 参与者信息同步

1. **添加第二个用户**
   - 用户 B 登录系统
   - 加入用户 A 创建的会议

2. **验证视频窗口更新**
   - 用户 A 的视频窗口应该显示：

3. **验证状态同步**
   - 用户 B 点击静音
   - 用户 A 的视频窗口应该显示用户 B 的静音图标

#### 预期结果：

- ✅ 新成员加入时，视频窗口自动更新

3. **从主窗口发送消息**
   - 用户 B 在主会议窗口的聊天中发送消息
   - 用户 A 的聊天悬浮窗口应该显示新消息

#### 预期结果：

- ✅ 悬浮窗口发送的消息正确同步

## 已知限制

### 1. 视频流传递

**当前状态**: 
- ✅ 参与者信息可以同步
- ❌ 实际视频流无法传递到悬浮窗口

**原因**: MediaStream 对象无法通过 IPC 序列化

**表现**: 
- 视频窗口显示参与者头像
- 不显示实际视频画面

**未来改进方案**:
- 方案 A: 在悬浮窗口中重新建立 WebRTC 连接
- 方案 B: 使用 Canvas 捕获帧并传递
- 方案 C: 使用 SharedArrayBuffer（实验性）

### 2. 窗口拖动

**当前状态**: 
- ✅ 基础拖动功能已实现
- ⚠️ 可能不够流畅

**改进**: 
- 需要优化拖动算法
- 考虑添加窗口吸附功能

### 3. 边框位置

**当前状态**: 
- ✅ 边框显示在屏幕四角
- ⚠️ 无法精确匹配实际共享区域

**原因**: 
- 无法获取用户选择的具体共享区域边界
- 目前使用整个屏幕尺寸

**改进**: 
- 需要 Electron API 支持获取共享区域信息

## 故障排查

### 问题 1: 悬浮窗口没有出现

**可能原因**:
1. Electron API 未正确暴露
2. IPC 通信失败
3. 窗口创建失败

**排查步骤**:
1. 打开 Electron 开发者工具
2. 检查控制台错误
3. 检查 `window.electron` 对象是否存在
4. 检查 `window.electron.startScreenShareOverlay` 方法是否存在

**解决方案**:
```javascript
// 在浏览器控制台测试
console.log(window.electron)
console.log(window.electron.startScreenShareOverlay)
```

### 问题 2: 窗口位置不正确

**可能原因**:
1. 屏幕分辨率计算错误
2. 多显示器配置问题

**排查步骤**:
1. 检查 `screen.getPrimaryDisplay()` 返回值
2. 检查 `workAreaSize` 是否正确

**解决方案**:
- 在 `main.js` 中添加日志输出屏幕信息

### 问题 3: 拖动不工作

**可能原因**:
1. `moveWindow` IPC 处理器未注册
2. 窗口引用丢失

**排查步骤**:
1. 检查 `main.js` 中是否有 `move-window` 处理器
2. 检查窗口变量是否为 null

**解决方案**:
- 确保 IPC 处理器正确注册
- 添加空值检查

### 问题 4: 数据不同步

**可能原因**:
1. 同步定时器未启动
2. IPC 消息未发送
3. 监听器未注册

**排查步骤**:
1. 检查 `startSyncToOverlay` 是否被调用
2. 检查控制台是否有同步日志
3. 检查 `window.electron.updateParticipants` 是否被调用

**解决方案**:
- 添加日志输出
- 检查定时器是否正常运行
